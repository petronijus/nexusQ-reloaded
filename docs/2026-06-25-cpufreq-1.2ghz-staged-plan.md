# Path B — TPS62361 DVFS to 1.2 GHz: staged, independently-validatable plan

**Goal:** scale the OMAP4460 MPU from the fixed 350 MHz up to 1.2 GHz on mainline
6.12, driving VDD_MPU through the external **TPS62361** like stock — done in small
stages, each of which builds, flashes, and is **validated pass/fail on hardware
before the next**. Voltage scaling can damage/destabilise the HW, so every stage
has explicit validation + rollback, and we never raise a voltage we haven't
proven.

Ground truth already established (see `2026-06-25-cpufreq-vdd-mpu-findings.md`):
this unit IS 1.2 GHz-fused (stock ran it), measured VDD_MPU table
**350/1025 · 700/1203 · 920/1317 · 1200/1380 mV**, ABB required at Nitro.

## Architecture decision

- **Transport:** stock drives VDD_MPU via the OMAP4 **VC (PRM SR-i2c)** hardware to
  TPS62361 slave 0x60, but ALSO registers the TPS on **i2c1** (`i2c_board_info`,
  bus 1). So the chip is on i2c1. **Plan A (preferred):** mainline `tps62360`
  regulator on `&i2c1` @ 0x60 + `cpufreq-dt` (sets voltage via i2c1 writes — the
  same software path stock uses for init). **Plan B (fallback):** if the chip's
  voltage SET is reachable only over the PRM VC/SR-i2c and NOT via the i2c1
  master, fall back to the OMAP voltage/VC layer (`ARM_OMAP2PLUS_CPUFREQ` +
  resurrected omap4 PMIC/VC binding). **Stage 0 decides which.**
- **Retasking:** on mainline VDD_MPU is on TWL6030 **VCORE1**; the TPS is dormant.
  To make the TPS drive VDD_MPU we must replicate `omap4_twl_tps62361_enable`
  (7 TWL6030 PREQ/SYSEN writes that pull VCORE1 off the MPU rail) — a custom
  board-init patch. Hand-over must be glitch-free: TPS already outputting ~1025 mV
  before VCORE1 is removed, or VDD_MPU droops and the CPU crashes.
- **ABB:** `abb_mpu` (`ti,abb-v2`) is already in the DT and brackets 1380 mV; keep
  it, verify `ti,abb_info` covers each OPP we enable.
- **Voltages:** use the **stock-measured nominals** (SmartReflex stays off, so no
  adaptive trim — never undervolt).

## Load-bearing unknowns to settle in Stage 0 (gate)
1. **Transport** — is TPS62361 voltage-SET reachable over the i2c1 software master,
   or only the PRM VC/SR-i2c? (Decides Plan A vs B.)
2. **VSEL strap** — which SET register (SET0..3) is hardware-active →
   `ti,vsel0-state-high` / `ti,vsel1-state-high` for the regulator binding. A wrong
   strap writes a non-active register → VDD_MPU stays low while freq rises → crash.

## Stages (each: change → build → flash → VALIDATE → rollback)

### Stage 0 — Resolve the two unknowns (READ-ONLY, no flash risk) — GATE
- **Do:** (a) search for a Nexus Q / steelhead schematic or teardown (TPS62361 i2c
  + VSEL strap). (b) Boot stock-adb (TPS is active + on i2c1 there) and read the
  TPS62361 SET0..SET3 registers @ 0x60 (i2cget if available, else expose via the
  stock regulator sysfs / a tiny read); the SET register whose value decodes to
  the current OPP voltage is the strapped/active one. (c) Confirm i2c1 reachability.
- **Validate:** we know, with evidence, (1) the transport and (2) the active SET
  register / strap. **Do not proceed without both.**
- **Risk:** none (read-only). **Rollback:** n/a.

### Stage 1 — Hand VDD_MPU to the TPS62361 at the BOOT OPP only (no scaling)
- **Do:** custom retasking patch (replicate `omap4_twl_tps62361_enable`) +
  `tps62361@60` DT node on `&i2c1` (regulator-name "vdd_mpu", strap from Stage 0,
  min 500000 / max 1380000, boot-on, always-on). `REGULATOR_TPS62360=y`. Keep the
  OPP at 350 MHz (no cpufreq yet, or OPP table = OPP50 only). Order: TPS up @
  ~1025 mV → then retask VCORE1 off the MPU rail.
- **Validate:** boots; `/sys/class/regulator/.../name` shows vdd_mpu(tps62361),
  its voltage reads ~1025 mV; the TPS answers i2c (proves transport); system is
  **stable at 350 MHz** under load (the hand-over didn't droop). ABB still fine.
- **Risk:** HIGH (the hand-over). **Rollback:** revert patch → VCORE1 drives MPU
  (known-good). Fastboot is always the escape.

### Stage 2 — cpufreq plumbing + OPP100 (700 MHz)
- **Do:** `&cpu0 { cpu-supply = <&tps62361>; }`; `CPUFREQ_DT=y`,
  `CPUFREQ_DT_PLATDEV=y`; OPP table = {350000/1025000, 700000/1203000}.
- **Validate:** `scaling_available_frequencies` = 350000 700000; force 700 MHz →
  `scaling_cur_freq`=700000 AND TPS voltage reads **1203 mV**; stress test (e.g.
  `yes>/dev/null &` ×2 + a build) stable, no errors; thermals sane
  (`/sys/class/thermal`). ← first real scaling.
- **Risk:** medium. **Rollback:** drop the 700 MHz OPP (back to Stage 1).

### Stage 3 — OPP-Turbo (920 MHz)
- **Do:** add 920000/1317000 to the OPP table; verify `abb_mpu` `ti,abb_info`
  handles 1317 mV (bypass/nominal at this point).
- **Validate:** scale to 920 MHz, TPS voltage **1317 mV**, stress-stable, thermals.
- **Risk:** medium (higher current; near VCORE-replacement's purpose). **Rollback:**
  drop the 920 MHz OPP.

### Stage 4 — Nitro gating + OPP-Nitro (1200 MHz)
- **Do:** read the MPU DPLL trim eFuse (CONTROL_STD_FUSE, the bit
  `omap4460_mpu_dpll_trim_override` checks) to gate; only if fused add
  1200000/1380000. Ensure `abb_mpu` forces FBB (ABB=1) at 1380 mV (the table lists
  1375000/1389000 ABB=1, brackets 1380000).
- **Validate:** scale to 1.2 GHz, TPS voltage **1380 mV**, ABB FBB engaged,
  prolonged stress stable, thermals under throttle point. ← the goal (1.2 GHz =
  3.4× the 350 MHz floor).
- **Risk:** highest (max voltage/current/heat). **Rollback:** drop the Nitro OPP →
  cap at 920 MHz (still a big win).

### Stage 5 — Governor, thermal throttling, finalize → v1.4
- **Do:** default governor (ondemand, matching stock) + verify `CPU_THERMAL`/
  `TI_THERMAL` throttling maps the OPPs; clean up; document.
- **Validate:** governor scales under load + idles down; throttles on heat;
  long-run (hours) stable. Then cut **v1.4** (eth + USB + cpufreq).

## Safety principles (every stage)
- Never raise a voltage not in the stock-measured table.
- TPS must be outputting the target before the rail depends on it.
- Validate at each OPP before adding the next — one increment at a time.
- Keep fastboot reachable (it's bootloader-level, our escape) and eth/ssh up.
- If any stage fails validation, roll back that stage and diagnose before retrying.

## Stage 0 RESULTS (resolved 2026-06-25, authoritative from THIS unit's stock)

Read the live OMAP4 PRM_VC registers on our own unit via a pushed `peek` /dev/mem
reader on stock-adb (raw dump in scratchpad `prm_vc_dump.txt`), decoded against the
live anchor (MPU at 350 MHz, `curr_vp_volt` = 840000 µV):

- `PRM_VC_SMPS_SA` (0x4A307B88) = **0x00601212** → VDD_MPU SMPS **slave = 0x60**
  (TPS62361); VDD_CORE/VDD_IVA slave = 0x12 (TWL6030 SR).
- `PRM_VC_VAL_SMPS_RA_VOL` (0x4A307B8C) = **0x00015b55** → VDD_MPU voltage register
  = **0x01 (TPS62361 SET1)**; CORE=0x55, IVA=0x5b (TWL6030 VCORE regs). MPU is the
  byte that pairs with slave 0x60; 0x5b is not a valid TPS register, so MPU=0x01.
- `PRM_VC_VAL_CMD_VDD_MPU_L` (0x4A307B98) = **0x22222100** → ON vsel = 0x22 =
  500000 + 0x22·10000 = **840000 µV**, which equals the live `curr_vp_volt` → the
  vsel law **vsel = (uV − 500000) / 10000** is confirmed on this unit
  (1025→0x35, 1203→0x46, 1317→0x52, 1380→0x58).

**Resolved unknowns:**
1. **Transport:** the TPS62361 is on the **SR-i2c / VC path ONLY** — there is NO
   `*-0060` i2c device on stock (i2c only carries the TWL6030 group + TAS5713), and
   the SR-i2c is write-only. → **Plan A (tps62360 regulator on i2c1) is impossible;
   we MUST use Plan B: the OMAP4 `omap_voltage`/VC layer** (add an omap4 TPS62361
   PMIC description, slave 0x60 / volt-reg SET1 0x01 / the vsel law above, and drive
   it via the VC like stock). This is the bigger effort the plan flagged.
2. **Strap / SET register:** TPS62361 **SET1 (0x01)** is the active register
   (`ti,vsel0-state-high`, vsel1 low) — but since we use the VC, the value goes into
   the omap_voltage PMIC `volt_reg_addr = 0x01`, not a tps62360 DT node.

**Cross-check note (Petr's principle):** the PandaBoard-ES reference drives the
TPS62361 VSEL0 via an OMAP GPIO (GPIO_WK7); our Nexus Q **straps SET1** instead —
DIFFERENT. Trusting the internet reference would have mis-set VDD_MPU. The value
above comes from THIS unit's live PRM_VC, not a generic datasheet.

**Plan revision:** Stage 1 becomes "resurrect/add the OMAP4 omap_voltage + VC +
TPS62361 PMIC description (slave 0x60, volt-reg 0x01, vsel law) and enable
ARM_OMAP2PLUS_CPUFREQ, capped at the boot OPP" — a larger kernel change than the
DT-regulator approach. Stages 2-4 (add 700/920/1200 MHz) unchanged.
