# Path B — TPS62361 DVFS to 1.2 GHz: staged, independently-validatable plan

**Goal:** scale the OMAP4460 MPU from the fixed 350 MHz up to 1.2 GHz on mainline
6.12, driving VDD_MPU through the external **TPS62361** like stock — done in small
stages, each of which builds, flashes, and is **validated pass/fail on hardware
before the next**. Voltage scaling can damage/destabilise the HW, so every stage
has explicit validation + rollback, and we never raise a voltage we haven't
proven.

Ground truth already established (see `2026-06-25-cpufreq-vdd-mpu-findings.md`):
this unit IS 1.2 GHz-fused (stock ran it), measured VDD_MPU table
**350/1025 · 700/1203 · 920/1317 · 1200/1380 mV**, ABB required at Nitro.

## Architecture decision

- **Transport:** stock drives VDD_MPU via the OMAP4 **VC (PRM SR-i2c)** hardware to
  TPS62361 slave 0x60, but ALSO registers the TPS on **i2c1** (`i2c_board_info`,
  bus 1). So the chip is on i2c1. **Plan A (preferred):** mainline `tps62360`
  regulator on `&i2c1` @ 0x60 + `cpufreq-dt` (sets voltage via i2c1 writes — the
  same software path stock uses for init). **Plan B (fallback):** if the chip's
  voltage SET is reachable only over the PRM VC/SR-i2c and NOT via the i2c1
  master, fall back to the OMAP voltage/VC layer (`ARM_OMAP2PLUS_CPUFREQ` +
  resurrected omap4 PMIC/VC binding). **Stage 0 decides which.**
- **Retasking:** on mainline VDD_MPU is on TWL6030 **VCORE1**; the TPS is dormant.
  To make the TPS drive VDD_MPU we must replicate `omap4_twl_tps62361_enable`
  (7 TWL6030 PREQ/SYSEN writes that pull VCORE1 off the MPU rail) — a custom
  board-init patch. Hand-over must be glitch-free: TPS already outputting ~1025 mV
  before VCORE1 is removed, or VDD_MPU droops and the CPU crashes.
- **ABB:** `abb_mpu` (`ti,abb-v2`) is already in the DT and brackets 1380 mV; keep
  it, verify `ti,abb_info` covers each OPP we enable.
- **Voltages:** use the **stock-measured nominals** (SmartReflex stays off, so no
  adaptive trim — never undervolt).

## Load-bearing unknowns to settle in Stage 0 (gate)
1. **Transport** — is TPS62361 voltage-SET reachable over the i2c1 software master,
   or only the PRM VC/SR-i2c? (Decides Plan A vs B.)
2. **VSEL strap** — which SET register (SET0..3) is hardware-active →
   `ti,vsel0-state-high` / `ti,vsel1-state-high` for the regulator binding. A wrong
   strap writes a non-active register → VDD_MPU stays low while freq rises → crash.

## Stages (each: change → build → flash → VALIDATE → rollback)

### Stage 0 — Resolve the two unknowns (READ-ONLY, no flash risk) — GATE
- **Do:** (a) search for a Nexus Q / steelhead schematic or teardown (TPS62361 i2c
  + VSEL strap). (b) Boot stock-adb (TPS is active + on i2c1 there) and read the
  TPS62361 SET0..SET3 registers @ 0x60 (i2cget if available, else expose via the
  stock regulator sysfs / a tiny read); the SET register whose value decodes to
  the current OPP voltage is the strapped/active one. (c) Confirm i2c1 reachability.
- **Validate:** we know, with evidence, (1) the transport and (2) the active SET
  register / strap. **Do not proceed without both.**
- **Risk:** none (read-only). **Rollback:** n/a.

### Stage 1 — Hand VDD_MPU to the TPS62361 at the BOOT OPP only (no scaling)
- **Do:** custom retasking patch (replicate `omap4_twl_tps62361_enable`) +
  `tps62361@60` DT node on `&i2c1` (regulator-name "vdd_mpu", strap from Stage 0,
  min 500000 / max 1380000, boot-on, always-on). `REGULATOR_TPS62360=y`. Keep the
  OPP at 350 MHz (no cpufreq yet, or OPP table = OPP50 only). Order: TPS up @
  ~1025 mV → then retask VCORE1 off the MPU rail.
- **Validate:** boots; `/sys/class/regulator/.../name` shows vdd_mpu(tps62361),
  its voltage reads ~1025 mV; the TPS answers i2c (proves transport); system is
  **stable at 350 MHz** under load (the hand-over didn't droop). ABB still fine.
- **Risk:** HIGH (the hand-over). **Rollback:** revert patch → VCORE1 drives MPU
  (known-good). Fastboot is always the escape.

### Stage 2 — cpufreq plumbing + OPP100 (700 MHz)
- **Do:** `&cpu0 { cpu-supply = <&tps62361>; }`; `CPUFREQ_DT=y`,
  `CPUFREQ_DT_PLATDEV=y`; OPP table = {350000/1025000, 700000/1203000}.
- **Validate:** `scaling_available_frequencies` = 350000 700000; force 700 MHz →
  `scaling_cur_freq`=700000 AND TPS voltage reads **1203 mV**; stress test (e.g.
  `yes>/dev/null &` ×2 + a build) stable, no errors; thermals sane
  (`/sys/class/thermal`). ← first real scaling.
- **Risk:** medium. **Rollback:** drop the 700 MHz OPP (back to Stage 1).

### Stage 3 — OPP-Turbo (920 MHz)
- **Do:** add 920000/1317000 to the OPP table; verify `abb_mpu` `ti,abb_info`
  handles 1317 mV (bypass/nominal at this point).
- **Validate:** scale to 920 MHz, TPS voltage **1317 mV**, stress-stable, thermals.
- **Risk:** medium (higher current; near VCORE-replacement's purpose). **Rollback:**
  drop the 920 MHz OPP.

### Stage 4 — Nitro gating + OPP-Nitro (1200 MHz)
- **Do:** read the MPU DPLL trim eFuse (CONTROL_STD_FUSE, the bit
  `omap4460_mpu_dpll_trim_override` checks) to gate; only if fused add
  1200000/1380000. Ensure `abb_mpu` forces FBB (ABB=1) at 1380 mV (the table lists
  1375000/1389000 ABB=1, brackets 1380000).
- **Validate:** scale to 1.2 GHz, TPS voltage **1380 mV**, ABB FBB engaged,
  prolonged stress stable, thermals under throttle point. ← the goal (1.2 GHz =
  3.4× the 350 MHz floor).
- **Risk:** highest (max voltage/current/heat). **Rollback:** drop the Nitro OPP →
  cap at 920 MHz (still a big win).

### Stage 5 — Governor, thermal throttling, finalize → v1.4
- **Do:** default governor (ondemand, matching stock) + verify `CPU_THERMAL`/
  `TI_THERMAL` throttling maps the OPPs; clean up; document.
- **Validate:** governor scales under load + idles down; throttles on heat;
  long-run (hours) stable. Then cut **v1.4** (eth + USB + cpufreq).

## Safety principles (every stage)
- Never raise a voltage not in the stock-measured table.
- TPS must be outputting the target before the rail depends on it.
- Validate at each OPP before adding the next — one increment at a time.
- Keep fastboot reachable (it's bootloader-level, our escape) and eth/ssh up.
- If any stage fails validation, roll back that stage and diagnose before retrying.

## Stage 0 RESULTS (resolved 2026-06-25, authoritative from THIS unit's stock)

Read the live OMAP4 PRM_VC registers on our own unit via a pushed `peek` /dev/mem
reader on stock-adb (raw dump in scratchpad `prm_vc_dump.txt`), decoded against the
live anchor (MPU at 350 MHz, `curr_vp_volt` = 840000 µV):

- `PRM_VC_SMPS_SA` (0x4A307B88) = **0x00601212** → VDD_MPU SMPS **slave = 0x60**
  (TPS62361); VDD_CORE/VDD_IVA slave = 0x12 (TWL6030 SR).
- `PRM_VC_VAL_SMPS_RA_VOL` (0x4A307B8C) = **0x00015b55** → VDD_MPU voltage register
  = **0x01 (TPS62361 SET1)**; CORE=0x55, IVA=0x5b (TWL6030 VCORE regs). MPU is the
  byte that pairs with slave 0x60; 0x5b is not a valid TPS register, so MPU=0x01.
- `PRM_VC_VAL_CMD_VDD_MPU_L` (0x4A307B98) = **0x22222100** → ON vsel = 0x22 =
  500000 + 0x22·10000 = **840000 µV**, which equals the live `curr_vp_volt` → the
  vsel law **vsel = (uV − 500000) / 10000** is confirmed on this unit
  (1025→0x35, 1203→0x46, 1317→0x52, 1380→0x58).

**Resolved unknowns:**
1. **Transport:** the TPS62361 is on the **SR-i2c / VC path ONLY** — there is NO
   `*-0060` i2c device on stock (i2c only carries the TWL6030 group + TAS5713), and
   the SR-i2c is write-only. → **Plan A (tps62360 regulator on i2c1) is impossible;
   we MUST use Plan B: the OMAP4 `omap_voltage`/VC layer** (add an omap4 TPS62361
   PMIC description, slave 0x60 / volt-reg SET1 0x01 / the vsel law above, and drive
   it via the VC like stock). This is the bigger effort the plan flagged.
2. **Strap / SET register:** TPS62361 **SET1 (0x01)** is the active register
   (`ti,vsel0-state-high`, vsel1 low) — but since we use the VC, the value goes into
   the omap_voltage PMIC `volt_reg_addr = 0x01`, not a tps62360 DT node.

**Cross-check note (Petr's principle):** the PandaBoard-ES reference drives the
TPS62361 VSEL0 via an OMAP GPIO (GPIO_WK7); our Nexus Q **straps SET1** instead —
DIFFERENT. Trusting the internet reference would have mis-set VDD_MPU. The value
above comes from THIS unit's live PRM_VC, not a generic datasheet.

**Plan revision:** Stage 1 becomes "resurrect/add the OMAP4 omap_voltage + VC +
TPS62361 PMIC description (slave 0x60, volt-reg 0x01, vsel law) and enable
ARM_OMAP2PLUS_CPUFREQ, capped at the boot OPP" — a larger kernel change than the
DT-regulator approach. Stages 2-4 (add 700/920/1200 MHz) unchanged.

## Stage 1 DESIGN (2026-06-25) — the VC↔regulator bridge

Mainline 6.12 mach-omap2 still contains the whole OMAP4 voltage stack
(`voltage.c`, `vc.c`, `vp.c`, `vc44xx_data.c`, `vp44xx_data.c`,
`voltagedomains44xx_data.c`, `omap_twl.c`, `omap-cpufreq.c`,
`omap_voltage_register_pmic()`), and it IS partly alive on a DT boot:
- `io.c:602 omap44xx_voltagedomains_init()` registers the omap4 voltage domains.
- `pm.c:148 omap_voltage_late_init()` → `omap_vc_init_channel()` and sets
  `voltdm->scale = omap_vc_bypass_scale / omap_vp_forceupdate_scale` (voltage.c:262)
  — the function that writes the VC → SR-i2c → PMIC.

**What's missing / wrong for us:**
1. `omap4_twl_init()` (omap_twl.c:213, registers the omap4 PMICs and lets
   `omap_voltage_late_init` wire the VC channel) is **never called** — only
   `omap3_twl_init()` is (pm.c:145). So on omap4 no PMIC is registered → the VC
   channel for VDD_MPU is not configured.
2. The existing `omap4_mpu_pmic` (omap_twl.c:159) is the **TWL6030** config
   (`OMAP4_SRI2C_SLAVE_ADDR` 0x12, `twl6030_vsel_to_uv`). We need the **TPS62361**
   variant for VDD_MPU: slave **0x60**, volt-reg **SET1 (0x01)**, vsel law
   `(uV−500000)/10000` (Stage 0 values). VDD_CORE/VDD_IVA stay TWL6030.
3. `omap-cpufreq.c` (ARM_OMAP2PLUS_CPUFREQ) scales via a Linux **regulator**
   (`regulator_get(mpu_dev,"vcc")` + `regulator_set_voltage`, line 161/78) — it does
   NOT call the omap_voltage layer. The TPS62361 has **no Linux regulator** (SR-i2c
   only). So there is a gap: cpufreq speaks "regulator", the TPS speaks "VC".

**Chosen approach — a thin VC-bridge regulator:** add a small regulator driver whose
`set_voltage(uV)` calls the omap_voltage scale path
(`omap_voltage_scale_voltage(mpu_voltdm, uV)` → `voltdm->scale` →
`omap_vc_bypass_scale` → PRM_VC bypass write to slave 0x60 / reg 0x01 / vsel). Register
it as the MPU cpu-supply ("vcc"/`cpu-supply`). Then either omap-cpufreq.c or cpufreq-dt
drives it. This reuses the intact VC code instead of resurrecting a full PMIC/i2c stack,
and is the minimal bridge.

### Stage 1 sub-stages (each validatable)
- **1a — Wire the omap4 voltage layer for the TPS62361 (no scaling, no retasking).**
  Add `omap4460_mpu_pmic` (TPS62361: slave 0x60, volt-reg 0x01, tps6236x vsel funcs)
  + call `omap4_twl_init()` (MPU→TPS variant) for omap4460. Validate: boots; the VC
  channel for vdd_mpu is configured (debug read of PRM_VC_SMPS_SA/RA_VOL shows
  0x60/0x01 set by US, not the bootloader); VDD_MPU still physically on VCORE1 (we
  haven't retasked) so the system is unchanged/stable. ← proves the registration
  without touching the rail.
- **1b — The VC-bridge regulator + the retasking, at the boot OPP only.** Add the
  bridge regulator (set_voltage→omap_voltage_scale) and the `omap4_twl_tps62361_enable`
  retasking (VCORE1→TPS) ordered safely (TPS commanded to ~1025 mV via the VC first,
  then VCORE1 pulled off). Validate: VDD_MPU now driven by the TPS (read the VC/VP
  voltage), system **stable at 350 MHz** under load. HIGH risk; rollback = revert.
- **1c — cpufreq wiring at OPP50 only.** cpu-supply = bridge regulator, enable
  `ARM_OMAP2PLUS_CPUFREQ` (or cpufreq-dt) with OPP table = {350000}. Validate:
  cpufreq device exists, single OPP, voltage reads 1025 mV, stable. Then → Stage 2.

Config to add: `ARM_OMAP2PLUS_CPUFREQ=y` (or keep cpufreq-dt), and verify the
voltage/VC/VP layer is built (it is, in mach-omap2). ABB (`abb_mpu`) already in DT.

## Stage 1b — STOCK RETASKING DECODE (auditor, 2026-06-25) — authoritative

Full evidence-backed disassembly of stock `omap4_twl_tps62361_enable` from
`reverse-eng/vmlinux.bin`. **This is the spec for the 1b patch — replicate exactly.**

**Function:** `omap4_twl_tps62361_enable` @ **0xc0011e4c** (the old note 0xc0011f80 is a
*label inside* it = start of the TWL block; true prologue 0xc0011e4c, spans ..0xc00120d4).
It is the `.tps62361_enable` callback in the mpu PMIC descriptor (`tps_data @ 0xc003ebf8`).

**Two helpers it calls:**
- `omap_vc_bypass_send_i2c_msg(voltdm, slave, reg, data)` @ 0xc006dbbc — the **PRM VC /
  SR-i2c bypass** write to the TPS62361 (packs slave/reg/data into PRM_VC_VAL_BYPASS).
- `twl_modify(mask, value, reg)` @ 0xc0011df4 — RMW on **TWL6030 module 0xd**:
  `new = (old & ~mask) | (value & mask)` via twl_i2c_read/write_u8.

### The exact ordered sequence — TPS-first, guarded

**Phase A — program TPS62361 over VC bypass (slave 0x60) FIRST. If ANY of A1–A4
returns non-zero → error path, TWL6030 is NEVER touched (glitch-free guarantee):**

| # | TPS reg | data | meaning |
|---|---|---|---|
| A1 | SET1 = **0x01** | `uv_to_vsel(on_volt)` = **0x35** (~1025 mV) | command TPS to boot V before rail handover |
| A2 | RMP_CTRL = **0x06** | `(ramp_code<<5) | 6`, ramp_code clz-derived from on_volt clamped to 32000 (0x7d00) | ramp-rate **[EXACT BYTE TBD — auditor follow-up]** |
| A3 | CONTROL = **0x04** | strap-conditioned (gpio_vsel0==-1, [tps_data+8]) | SET-pin/PWM **[EXACT BYTE TBD]** |
| A4 | TEMP = **0x05** | **0** | control clear |

**Phase B — retask TWL6030 (module 0xd) so VCORE1 stops driving VDD_MPU. Runs only
if Phase A fully succeeded:**

| # | TWL6030 reg | value | mask | semantics (from fn's own pr_err strings) |
|---|---|---|---|---|
| B1 | 0xB4 | **[TBD: bit 0x20 set?]** | 0x20 | map APE PREQ1 |
| B2 | 0xB4 | 0x01 | 0x03 | sysen active |
| B3 | 0xB4 | 0x00 | 0x0C | sysen sleep |
| B4 | 0xB4 | 0x00 | 0x30 | sysen off |
| B5 | 0xB3 | 0x01 | 0x07 | map APE SYEN |
| B6 | 0x5E | 0x00 | 0x01 | unmap APE VCORE3 |
| B7 | 0x64 | 0x00 | 0x01 | unmap APE VMEM |

0xB3/0xB4 = TWL6030 **SYSEN** resource CFG_TRANS/CFG_STATE; 0x5E/0x64 = VCORE3/VMEM
PREQ1 resource-assignment bytes. Net: VCORE1 group off VDD_MPU, external TPS (gated by
the SYSEN resource) takes over.

**TPS enable mechanism:** NO OMAP GPIO, NO TWL regulator. Enabled via the **TWL6030
SYSEN resource** (active-in-APE by B2/B5). VSEL straps = **-1** (no pins; SET1 over VC).
Confirmed in `tps_data` (ff ff ff ff = -1) + call site `tps6236x_init(r2=-1,[sp]=-1)`.

**Call site:** board `.init_machine` (arch initcall, after `omap4_twl_init`),
**soc_is_omap446x-gated** (`omap_rev()>>20 == 0x443/0x446`). SmartReflex/VP NOT required
(uses VC bypass, not VP force-update). `tps6236x_init` @ 0xc001216c writes a timing/ramp
word pair to **0xc0740d80** (two const pairs by a strap bit) — **[identity + whether
needed = TBD auditor follow-up]**.

### Mainline mechanism (what we have / must add)
- `omap_vc_bypass_scale` (vc.c:191) exists — writes vsel into SET1 (`vc->volt_reg_addr`)
  + slave, valid+poll. Packs `(vsel<<data_shift)|(reg<<regaddr_shift)|(slave<<slaveaddr_shift)`
  into `bypass_val_reg` (PRM_VC_VAL_BYPASS). It does NOT take an arbitrary reg.
- **MUST ADD** `omap_vc_bypass_send_i2c_msg(voltdm, slave, reg, data)` — factor the core
  of omap_vc_bypass_scale (vc.c:204–233), parameterized by reg+data, **no** pre/post-scale
  (those are voltage bookkeeping; raw register writes like ramp/ctrl must not touch it).
  Export it + declare in vc.h. Use for ALL of Phase A.
- Phase B: `twl_i2c_read_u8`/`twl_i2c_write_u8` on **TWL6030 module** (already used by 1a
  omap_twl.c) for the RMW. (`TWL_MODULE_PM_RECEIVER` = module 0xd region.)

### CRITICAL constraints (auditor)
1. Stage 1a only pointed the VC channel at slave 0x60 — **VDD_MPU is still physically on
   TWL6030 VCORE1**. Until the retasking lands, issuing ANY MPU VC scale writes the TPS
   while VCORE1 still drives the rail → **contention** (the write is cosmetic / risky).
   Keep MPU cpufreq OFF until 1b (it is — `ARM_OMAP2PLUS_CPUFREQ` off).
2. For 1b we have no MPU OPP/on_volt in mainline → **hardcode the boot voltage**:
   `tps62361_uv_to_vsel(1025000)` = 0x35 for A1 (use the helper from patch 0013). Full OPP
   table comes in Stage 2.
3. `abb_mpu` has **no DTS node** — out of scope for the 1025 mV handover, but **audit
   before 1.2 GHz** (high OPP needs ABB FBB).
4. The one value to confirm LIVE before trusting in a handover: SET1 = **0x01** — already
   live-confirmed in Stage 0 (PRM_VC_VAL_SMPS_RA_VOL = 0x00015b55 → MPU reg byte 0x01).
5. `CONFIG_REGULATOR_TPS62360` is irrelevant — it's an i2c regulator, can't reach the
   VC-only chip. Do not use it for this rail.

**1b patch will be:** (i) add `omap_vc_bypass_send_i2c_msg` to vc.c + vc.h; (ii) add
`omap4460_vdd_mpu_retask()` replicating Phase A (A1=0x35, A2/A3 exact, A4=0) then Phase B
(B1–B7), guarded; (iii) call it soc_is_omap446x-gated after omap4_twl_init +
omap_voltage_late_init (so `voltdm->scale`/VC channel ready). Validate: stable @ 350 MHz
under load (handover didn't droop), VC/VP reads the TPS.

### Stage 1b — FINAL evaluated values (auditor round 2, 2026-06-25) — supersedes TBDs above

Two CORRECTIONS to the round-1 table, then the exact bytes:

- **`on_volt` (handover target) — stock = 1410000 µV (1.41 V), vsel 0x5B — NOT 1025 mV.**
  Stock commands the TPS to 1.41 V at A1 because that is the voltage **its** bootloader
  left VDD_MPU at, then DVFS scales down. The faithful principle is "command the TPS to
  the CURRENT rail voltage so the SYSEN handover is a no-step transition." **OUR** rail at
  the handover is the 350 MHz boot OPP on VCORE1 = **1025 mV**, so **WE program vsel 0x35
  (1025 mV)** — matching our rail, not stock's 1.41 V. (No post-scale needed; we're already
  at steady-state OPP voltage. To be maximally seamless we could read the live VCORE1 SMPS
  voltage and match it exactly; 1025 mV is the known boot value.) Stock's hard TPS ceiling
  is 1770000; our patch-0013 vddmax 1390000 is fine since our OPP range tops at Nitro 1380.
- **VSEL0 is a REAL OMAP GPIO (GPIO 7, pinmux 0xe00) — NOT strapped.** The round-1 "both
  VSEL strapped -1" was WRONG: stock `tps6236x_init(arg0=1, vsel0=7, vsel1=-1)` requests
  **GPIO 7 for VSEL0** (VSEL1 strapped low). The TPS picks its active SET register from
  VSEL0/1; **for the VC's SET1 (reg 0x01) writes to actually drive the output, VSEL0 must
  select SET1.** Stage 0 confirmed SET1 is active on stock → GPIO 7 is in the SET1-selecting
  state on stock. ⚠️ **CRITICAL OPEN ITEM for 1b: ensure GPIO 7 selects SET1 on our mainline
  before the handover** (identify the OMAP gpio#/pad, read its live state, drive it if the
  bootloader didn't). If VSEL0 is wrong, our SET1 writes hit an inactive register → VDD_MPU
  is whatever the *other* SET reg holds → droop/crash. **Resolve before flashing 1b.**
  **LIVE-resolved:** GPIO 7 = **gpio_wk7 = mainline gpio 519** (gpiochip0 @ 4a310000, base
  512, line 7 — same as PandaBoard GPIO_WK7). On our 1a it is **NOT requested/configured**
  → VSEL0 undefined. Must mux the gpio_wk7 wkup pad (MUX_MODE3) + drive it (HIGH=SET1
  expected) before handover. Our DTS already muxes gpio_wk6 (IOPAD 0x050) + gpio_wk1 (0x042).
  Exact pad offset + drive value pending auditor round 3.
  (The strap *config fields* tps_data+4/+8 stay -1 either way — that's what the enable fn
  branches on — so the enable register stream is unchanged; but the physical pad still must
  select SET1.)

### Stage 1b — first flash result + the timing fix (2026-06-25/26)

Flashed boot-cpufreq-1b.img. Booted **safe + stable** (gpio-519 hogged out-hi ✓, Phase A
succeeded), BUT Phase B failed: `vdd_mpu retask: TWL6030 retask write failed (-1)` because
`twl: not initialized` at our call point (0.75 s). Root cause from dmesg: the **OMAP i2c
controllers probe ASYNC ~3-4 s** (i2c-2 @2.945 s, i2c-0/the TWL bus @3.35 s) — long after
`omap2_common_pm_late_init`. So the rail **stayed on VCORE1** (no droop — the guard left it
safe). The VC/Phase-A side IS ready early (omap_voltage_late_init); only Phase B needs i2c.

**Fix:** defer the hand-over — `omap4460_vdd_mpu_retask()` now `schedule_delayed_work()`s a
poll (`twl_i2c_read_u8` reg 0x00 every 500 ms, ≤60 tries) that runs the retask once the
TWL6030 answers. (Stock is non-DT and brings the TWL up synchronously before its board
init_machine retask call, so it needs no poll; our poll re-establishes that "after-TWL-ready"
ordering under mainline's async DT probe.)
Re-spun 0014 (adds `#include <linux/workqueue.h>`, splits `omap4460_vdd_mpu_do_retask()` +
the poll work); applies clean.

**Auditor CONFIRMED (2026-06-26):** stock relies PURELY on synchronous non-DT board-file
ordering — TWL6030 probes at subsys_initcall (i2c-omap), done before `steelhead_init`
init_machine (end of do_initcalls); the enable bails on any not-ready twl read as a HARD
ERROR (no poll/retry/defer anywhere — `[twl_priv+0xe8]` ready flag, not-init error
@0xc02ae7f8). Our poll-until-ready is the CORRECT adaptation of that invariant for mainline's
async DT probe — not mirroring a stock mechanism (stock needs none). For raw twl_i2c register
access the poll has no partial-readiness race (once `ready=1` all raw writes work). Guard
semantics (Phase A→B, abort-on-error, poll gates the whole enable) match stock. → pkgrel 9,
rebuild + reflash + validate handover completes.

### Stage 1b — round 3 CONFIRMED (VSEL0 GPIO) + implementation details (2026-06-25)

**VSEL0 GPIO — from OUR stock (NOT PandaBoard; PandaBoard only incidentally agrees):**
- "GPIO 7" = **gpio_wk7**, pad **`fref_clk4_req`** mode 3. Source: our stock `omap_mux` table
  @ 0xc0043764 (`reg_offset=0x005A, gpio=7`, muxnames `[fref_clk4_req, fref_clk5_out, NULL,
  gpio_wk7]`). Live device = gpio **519** (gpiochip0 base 512 + 7), currently unconfigured.
- Stock **drives it HIGH**: `omap_mux_init_gpio(7,0xe00)` → `gpio_request(7)` →
  `gpio_direction_output(7, 1)` (value=1 decoded from the `(0xe00-0xe00==0)?1:0` ARM idiom @
  0xc0011d70-dac). VSEL0=1, VSEL1 strapped low → `{0,1}` = **SET1** active = matches Stage-0
  live PRM_VC (voltage→reg 0x01). Load-bearing: our SET1 writes only drive the output because
  VSEL0 is held high.
- DTS: pad = `OMAP4_IOPAD(0x05a, PIN_OUTPUT | MUX_MODE3)` under `&omap4_pmx_wkup`.

**TWL6030 module for Phase B RMW = `TWL6030_MODULE_ID0`** (subchip 0 = slave **0x48 POWER**;
1a omap_twl.c already uses ID0 for VCORE; regs 0x5E/0x64/0xB3/0xB4 are absolute addrs in 0x48).
