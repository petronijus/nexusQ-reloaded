# 2026-07-16 — v1.10.1 bug-fix release (five fixes)

A bug-fix release on top of v1.10.0. Five faults, each root-caused with evidence;
built, flashed, and hardware-verified on a fresh v1.10.1 flash.

**Shipped package set (v1.10.1):** kernel **r44**, device **r49**, nexusq-btagent
**r4**, nexusq-control r10 (unchanged), nexusq-setupd r4 (unchanged), nexusqd r10
(unchanged), firmware r2 (unchanged). App **1.3.1+9** (own version track — NOT part
of the image). Base: v1.10.0.

---

## 1. Factory WiFi MAC — FIXED (was a known-open issue since v1.9.0/v1.10.0)

**Kernel patch `0043-ARM-dts-omap4-steelhead-wifi-local-mac-address.patch` (r43 → r44)**
adds `local-mac-address = [f8 8f ca 20 48 e1]` to the DTS `wifi@1` node, mirroring the
BT `local-bd-address`.

**Hardware-verified on the fresh v1.10.1 flash:** `ethtool -P wlan0` now reports
PERMANENT address **`f8:8f:ca:20:48:e1`** (was the chip OTP MAC `14:7d:c5:3a:35:b5`,
Murata OUI).

**Stock-parity chain (why DT is the only route):**
- Stock got the WiFi MAC from the **bootloader cmdline** (`androidboot.wifi_macaddr=`,
  parsed by a board callback into the bcmdhd platform data), read from the per-device
  **efs/factory partition** — a path we cannot reproduce: our U-Boot does not pass it,
  and `CONFIG_CMDLINE_FORCE=y` would discard it anyway.
- The shipped nvram (`brcmfmac4330-sdio.txt`) carries only a **generic Broadcom
  placeholder** `macaddr=00:90:4c:c5:12:38`, and **brcmfmac ignores it** because the
  chip has a MAC burned in **OTP**. Verified live 2026-07-16: overriding the nvram
  macaddr and reloading the module left the permanent MAC at the OTP value.
- The **only** route is DT: `brcmfmac`'s `brcmf_of_probe()` reads `local-mac-address`
  via `of_get_mac_address()` into `settings->mac` and **programs it over OTP**, so the
  factory MAC becomes wlan0's permanent address at the driver level — no per-profile
  clone needed.

This also **closes the onboarding-profile gap**: the NetworkManager `cloned-mac-address`
pin only fixed the *baked* dev profile; the profile `nexusq-setupd` created during
onboarding fell back to OTP. With the MAC pinned at the driver, NM `permanent` == the
factory MAC on every profile, so setupd's created profile no longer needs a
`cloned-mac-address` pin. The factory MACs are consecutive: BT `…49:e5`, WiFi `…48:e1`.

> Lease-lookup guidance (updated): after a **v1.10.1+** flash, look wlan0 leases up by
> the factory MAC `f8:8f:ca:20:48:e1` again (and the router hostname is populated). On
> **≤ v1.10.0** images the on-air MAC is the OTP `14:7d:c5:3a:35:b5` with an empty
> hostname.

## 2. btagent fd leak → app "kept disconnecting" (btagent r3 → r4)

The phone's Devices screen showed BT calls failing every 3 s with
**"bluetooth agent unreachable: No such file or directory"** — while the connection
itself was healthy. On device, `nexusq-btagent` was `active` but **its control-socket
file was GONE**, and the journal repeated `[Errno 24] No file descriptors available`.

**Cause:** `start_control()` (opens a listening socket + a GLib watch) was called from
the **10 s `_tick`** as well as from `run()`, leaking one fd per tick until the process
exhausted them (~1024) and crashed mid-tick with the socket removed.

**Fix:** `_tick` no longer calls `start_control()`; `start_control()` is guarded
idempotent. **Verified:** fd count flat at 8 across ticks.

> This bug was found on the first try by the app's new in-app debug log (fix #5) — the
> phone-side record of the banner flapping every ~3 s pointed straight at the agent
> socket.

## 3. onboard SIGSEGV every boot (device r48 → r49)

The on-screen keyboard crashed in its native `osk` module on **every boot** — and an
on-screen keyboard is useless on an appliance with no touchscreen and no attached input.
Its autostart lives in **`/etc/xdg/lxqt-tablet/autostart/`**, NOT the plain `autostart/`
that our XDG_CONFIG_DIRS shadow covers, so the apk **trigger** now also fires there and
neuters onboard's own autostart file (`Hidden=true`) once present — the same trigger
mechanism already used for the pulseaudio mixer path, and certain because it edits the
exact file that gets read. Idempotent across onboard upgrades. **Verified:** 0 onboard
coredumps on the fresh v1.10.1 boot.

## 4. librespot boot-race storm (device r48 → r49, same package)

The wrapper waited **30 s** for wlan0's DHCP IPv4 before starting librespot; BCM4330
cold-boot association routinely takes longer, so the first start gave up → systemd
`Restart` → a **5× storm** until WiFi finally came up. The unit's
`After=network-online.target` does **not** help: at the USER-manager level that target
is not wired to real connectivity, so the wrapper's poll IS the gate. **Fix:** extended
the wait **30 → 180 s** so association is caught on the first process (exit 1 stays for a
genuinely dead radio). **Verified:** 0 restarts.

## 5. Companion app: debug mode + Devices red-bar flicker (app 1.2.0 → 1.3.1)

App own version track — **NOT** part of the image.

- **New "Debug mode" toggle** (Devices → Developer) reveals an **always-on** in-app
  connection log. **Collection is always on**; the toggle only reveals the viewer, so
  the history leading up to a flicker is already captured (a 600-entry ring of short
  strings). It records: the banner switch (connection UP/DOWN), DROP causes
  (peer-closed vs socket-error vs supervisor-disconnect on a failed probe), probe
  latency, call timeouts with pending-queue depth, slow/late responses, and lifecycle
  transitions (resumed/inactive/paused). **Method names only, never params** (`setWifi`
  carries the PSK). This log is what found bug #2 on the first try.
- **Devices red-bar flicker fixed:** the Devices screen's 3 s background poll now
  **logs** failures instead of flashing the red error bar; only user-initiated actions
  show a visible error.
